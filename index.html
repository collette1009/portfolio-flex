About

고객의 문제를 제품의 기회로 바꾸는 사람,
모르는 도메인을 고객 언어로 번역하는 사람

플렉스의 CX는 단순 CS가 아니라고 생각합니다. 고객사의 인사 제도와 운영 기준이 플렉스라는 제품 안에서 안정적으로 작동하도록 돕는 일에 가깝다고 보았습니다. 저는 고객의 반복 문의에서 패턴을 읽고, 그것을 제품·운영 개선 과제로 바꾸는 방식으로 일해왔습니다.

프로모션 오인 VOC 173건을 전수 분류해 82%가 동일 구조 원인임을 데이터로 확인했고, 이를 바탕으로 운영 기준을 수립했습니다. 배송 VOC 패턴에서는 UX 구조 문제를 재정의해 CTO에게 직접 제안했고, 당일 구현까지 이끌었습니다. 그 결과 반복 CS 인입은 0건으로 줄었습니다.

HR·노무 도메인이 처음이라는 것을 알고 있습니다. 하지만 저는 새로운 도메인 앞에서 멈추지 않습니다. 택배 표준약관과 물류 계약서도 처음이었지만, Claude로 쟁점을 정리하고 NotebookLM으로 원문을 교차 확인해 정산·배상 판단 근거를 만들었습니다. CS 판단 기준이 없던 환경에서는 처리 기준과 에스컬레이션 구조를 직접 설계해 경영진 보고 후 전사 기준으로 확정했습니다.

어려운 지식을 빠르게 파악해 고객이 이해할 수 있는 언어로 바꾸는 것. 그것이 제가 새로운 도메인에 들어가는 방식입니다.

AI 도구도 단순 보조 수단이 아니라 실무 파트너로 활용해왔습니다. 비개발 직군 중 유일하게 AI 업무비를 공식 승인받았고, Chrome Extension을 직접 구현해 반복 업무 동선을 8단계에서 2단계로 줄였습니다. 플렉스에서도 AI를 실제 CX 운영 개선에 붙여본 사람으로 기여하고 싶습니다.

배송 제약 상품(특정 요일 도착)이 장바구니에서 일반 품절로 표시되어 고객 문의가 일 6~10건 반복됐습니다. 단순 보고가 아니라 "시스템에서 품절과 배송일 제약을 구분할 수 있다면, 장바구니 상품 옆에 문구만 다르게 표시하는 MVP가 가능할까요?" — UX 솔루션 방향과 구체적인 예시 문구까지 직접 설계해 CTO에게 제안했습니다. CTO가 즉시 수용해 개발팀에 지시, 당일 구현됐고 반복 문의는 0건이 됐습니다. 이 subtext 구조는 이후 상품운영팀이 프로모션 안내에도 자체 확대 적용, 단일 제안이 사내 개선 표준으로 전파됐습니다.

AI로 계약·약관 쟁점 정리 → 원문 교차 확인 · 배상회수율 14.7배 개선

택배 표준약관과 물류 계약서를 처음부터 다시 확인해, 외부 파트너 배상 구조를 재정의한 사례입니다. Claude로 계약 쟁점을 정리하고, NotebookLM으로 원문 근거를 교차 확인해 실제 청구 가능한 책임 구간을 찾았습니다. 숨겨져 있던 선공제 조항을 근거로 피해 기준과 정산 체계를 직접 설계했고, CEO 보고 후 전사 기준으로 확정했습니다. 처음 접하는 법·규정 기반 도메인도 같은 방식으로 학습하고, 고객이 이해할 수 있는 운영 기준으로 바꾸겠습니다.

배상회수율 14.7배 원문 교차 확인 → 협상 근거 → 월 1,900만 선공제 구조 확보

반복 입력과 기준값 오류가 누적되던 CS 처리 흐름을 Chrome Extension으로 직접 자동화했습니다. Claude를 활용해 로직을 설계·수정하고, 반복 CS 프로세스를 8단계에서 2단계로 줄였습니다. 자동화 범위는 반복 입력에 한정하고, 귀책 판단·최종 등록처럼 리스크가 있는 구간은 사람이 검수하도록 남겼습니다. 이 실무 활용성을 인정받아 비개발 직군 중 유일하게 AI 업무비를 공식 승인받았습니다.

문제 상황
배송 가능일이 다른 상품이 장바구니에 함께 담기면 주문이 차단됐지만, 고객에게는 그 이유가 충분히 안내되지 않았습니다. 고객은 이를 품절이나 시스템 오류로 인식했고, 동일한 문의가 일 6~10건 반복됐습니다.

핵심 문제는 고객의 이해 부족이 아니라, 주문이 막히는 시점에 시스템이 이유를 설명하지 못하는 UX 구조에 있었습니다.
실행 A — Fail-soft UX 설계
모든 예외 조건을 시스템에서 완벽히 분기하기는 어려웠기 때문에, API 응답값으로 판단 가능한 조건만 남겨 최소 실행 단위의 개선안을 설계했습니다.

고객이 주문 불가 이유를 이해하고 흐름을 이어갈 수 있도록 장바구니 내 subtext 안내 구조를 직접 기획했고, 데이터 자료와 함께 CTO에게 제안했습니다. 제안은 즉시 수용되어 당일 구현됐습니다.
실행 B — 운영 기준 확산
해당 subtext 구조는 이후 유사한 안내 케이스에도 적용 가능한 기준으로 확장됐습니다. 단순 문구 추가가 아니라, 고객이 막히는 시점에 필요한 정보를 제공하는 사내 UX 개선 기준으로 전파됐습니다.

문제 상황
골프장 400여 곳의 취소 정책·위약금 기준·동반자 변경 규정이 제각각이었습니다. 파트너사 어드민에는 데이터가 있었지만, 상담자가 바로 판단할 수 있는 기준으로 정리되어 있지 않아 동일 유형 문의에도 팀원마다 안내가 달라지는 상황이 발생했습니다. 오안내와 재문의가 반복되며 상담 품질 편차도 함께 커졌습니다.
실행
400여 개 제휴 골프장의 정책을 중요도순으로 재구조화하고, 예약·취소·위약·동반자 변경 기준을 유형별로 분류했습니다. 반복 문의가 발생하는 구간을 VOC 기준으로 다시 정리하고, 상담자가 동일 기준으로 판단할 수 있도록 운영 가이드와 Knowledge Base를 재정비했습니다.

정책 변경이 발생할 때는 어드민 기준이 최신 상태로 반영되도록 내부 공유 흐름을 정리했고, 신규 인력이 빠르게 기준을 익힐 수 있도록 온보딩 자료에도 반영했습니다.
결과
동일 유형 문의에 대한 상담 기준이 통일되었고, 정책 해석 차이로 인한 오안내와 재문의 가능성을 줄였습니다. 제휴사별 정책이 개인 숙련도에 의존하지 않고 팀 전체가 참고할 수 있는 기준으로 남게 되었으며, 신규 인력 온보딩 시간과 상담 품질 편차를 줄이는 데 기여했습니다.

문제 상황
외부 파트너 귀책으로 발생한 배송 사고와 물류 손실이 있었지만, 배상은 파트너사 관행에 따라 제한적으로 처리되고 있었습니다. 사고는 반복되는데 책임 기준과 비용 반영 구조가 명확하지 않아, 월 배상 회수액은 50만 원 수준에 머물러 있었습니다.
실행 A — 택배 표준약관 검토
택배 표준약관을 기준으로 파손·분실·배송 사고 시 청구 가능한 조건을 정리했습니다. Claude로 쟁점을 먼저 구조화하고, NotebookLM으로 원문 근거를 교차 확인해 사진·접수 기록 등 최소 증빙이 있을 경우 파트너에게 청구 가능한 기준을 세웠습니다. 기존 관행 중심의 처리 방식을 근거 기반 협상 구조로 전환했습니다.
실행 B — 물류 계약서 선공제 조항 확인
물류 계약서를 다시 검토하는 과정에서 직접 차감 가능한 선공제 조항을 확인했습니다. 해당 조항을 근거로 피해 유형, 비용 항목, 정산 기준을 재정리했고, CEO에게 보고해 전사 기준으로 확정했습니다.

문제 상황
대량 취소·반품·재배송 처리 과정에서 동일한 항목을 반복 입력해야 했습니다. 처리 건수가 늘어날수록 입력 피로도와 기준값 오류 가능성이 커졌고, 특히 귀책대상처럼 후속 정산과 연결되는 값은 한 번 잘못 입력되면 원인 분석과 비용 집계까지 왜곡될 수 있었습니다.
실행 A — 반복 입력 자동화
반복적으로 입력되는 필수값과 기준값을 먼저 분리했습니다. 이후 Chrome Extension을 직접 구현해 재배송·반품 등록 시 주요 항목이 단축키 한 번으로 자동 입력되도록 만들었습니다. 이를 통해 기존 8단계에 가까웠던 수동 입력 흐름을 2단계로 줄였습니다.
실행 B — 리스크 구간은 사람에게 남김
모든 단계를 자동화하지는 않았습니다. 귀책 판단, 예외 케이스 확인, 최종 등록처럼 비용·정산·고객 안내에 영향을 주는 구간은 담당자가 직접 확인하도록 남겼습니다. 자동화의 목적을 속도 향상이 아니라 반복 오류를 줄이고 기준값을 표준화하는 데 두었습니다.
실행 C — AI 업무 도구화
Claude를 로직 설계·수정·디버깅 보조에 활용했고, NotebookLM과 GPT는 VOC 분류, 계약·약관 원문 교차 확인, 운영 기준 정리에 사용했습니다. 단순히 AI 답변을 쓰는 것이 아니라, 운영 기준을 더 빠르고 정확하게 만드는 실무 파트너로 활용했습니다. 이 활용성을 인정받아 비개발 직군 중 유일하게 AI 업무비를 공식 승인받았습니다. 이 외에도 AI 기반 자금집행 리포트 자동화 구조를 선제 구축해 반복 보고 업무를 경영진 의사결정 참고 자료로 전환했습니다.
